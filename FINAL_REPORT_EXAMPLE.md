## Abstract

No-show appointments are a massive crisis continuing to face the NHS. This report investigates how
business intelligence can reduce the number of no-show appointments in the health service and
explores how a potential solution could help reduce no-show appointments.

The Doctor Book app is a new patient booking system that aims to provide a great user experience
allowing patients to book, cancel and edit their appointments wherever they are, whenever they
want. The report documents the full product life cycle from research to implementation and testing
of the product.

The hope is that the product created in this project, along with the data visualisations of no-show
appointments, could help prevent and reduce patients not showing up for their appointments.

Chapter 1 briefly overviews the project, including its aims and what will be delivered. Chapter 2
shows research taking place, and the user analysis in section 5 helps to inform the design of the app,
database and data visualisations in chapter 5. Chapter 3 explains the methodology driving the
project, while chapter 4 describes the potential risks and concerns and how these will be mitigated.
The implementation is then explored in chapter 7 before testing of the app occurs in chapter 8.
Chapter 9 evaluates the overall success and limitations of the project and product with a brief
description of what the developer has leant through the whole project. A conclusion then draws the
report to a close.

## 1. Introduction

```
Health services worldwide continue to struggle as the world emerges from the Covid- 19 pandemic.
In the UK, the NHS continues to struggle with a massive patient backlog, record waiting times and
health workers taking strike action. One of the major problems facing the NHS and GPs, in particular,
are no-show appointments, which are contributing heavily to the decline of staff and patients'
mental and physical health.
```
```
This project will explore the facts and figures behind no-show appointments through the use of data
analytics and visualisations to investigate why there are so many no-shows and the possible ways to
reduce these, along with creating a fit-for-purpose, proof-of-concept patient booking app, a possible
way to reduce these from occurring.
```
### 1.1 Project Background

```
Health Services within the UK are continually being pushed to the brim every day, with GP surgeries
among those facing the most pressure. With the overall number of GPs showing little growth from
2015 (BMA, 2023 ), GP appointments have only become more precious, especially as “bookings
reached record highs over the winter of 2021 with GPs seeing more patients than ever” (BMA,
2023). Currently, booking appointments are generally only available by ringing the surgery at 8 am.
However, this leads to lengthy phone queues and frustrated patients who cannot get an
appointment. The current model of ringing the surgery on a morning is not sustainable in today’s
world, as appointments should be easily accessible to all and readily available.
```
```
Additionally, more than 15 million GP appointments were missed in 2019 (NHS, 2019), resulting in
appointments being harder to find when you most need them. This is costing the NHS £216 million
in 2019 (NHS, 2019) but this money could have gone to better use with that appointment slot
potentially putting someone else’s life at risk.
```
### 1.2 Rationale

```
GP appointments should not be this hard to get a hold of, with me personally experiencing the stress
of trying to gain an appointment and knowing others who have had to go through this too. The
number of no-shows is way too high, with these appointments not only wasting GPs’ time but also
stopping people who really need them from speaking to a doctor, all while costing millions to the
taxpayer. It is, therefore, essential that a better way of booking appointments is found along with
methods of reducing the number of no-shows to help improve the health system for everyone.
```

Moreover, on a technologies front, this project will allow the developer to develop and enhance

## 2. Research: Literature Review and Analysis

### 2.1 User Experience / User Interface

```
User Experience (UX) will be a vital part of this project. User experience defines how well the user
can use the app from a design point of view. If an app is not designed well, it will likely lead to a poor
user experience. Therefore, the app design will be carefully thought about, including when
developing the initial wireframes. Mobile innovation company Clearbridge Mobile (no date) states
that “the UX determines if a user will return to your app or if they will delete it altogether”. This is,
therefore, an essential aspect to get right and determines how the user experience will act according
to the app's design and different elements of functionality.
```
```
“A good user experience is a must for modern software products” (Santoso and Schrepp, 2018). This
conference paper explains why UX is so important to developing apps in the 21st century as the
balance between requirements and design is often conflicted. Therefore, a middle ground needs to
be found to enhance the users' experience.
```
```
Adobe (no date) summaries how user interface “is an integral aspect of user experience” which
consists of two major design process: visual and interaction design. These are both critical to the
success of the app. The principles adobe sets out here will be taken forward into the design process.
```

#### 2.1.1 Principles of User Experience

```
A 2014 study led by Saucken et al. found that there were 20 user experience principles. The standout
principles included the following: “Prevent a bad first impression”, “Avoid the feeling of inability”,
“Enjoy a natural handling”, “Enhance controlled information”, “Natural usage for everyday life”,
“Enable individualisation”, “Minimise necessary user operations”.
```
```
Further standout principles from Career Foundry states the importance of having the right
Hierarchy for the app navigation and having a consistent design throughout the app. Career foundry
agrees with the 2014 study that the accessibility of the app is of the up most importance, giving user
controls and the right amount of confirmation (Grass, 2021). Costa (2021) further confirms this UX
principles list, summarising that “so many factors need to be considered”. A responsive design must
be achieved so the app can work on various devices. The typography, colours and layout need to be
the same.
```
```
Additionally, Idealogic concluded that poor navigation, poor app speed, excessive categorisation,
complication authentication procedures and no auto-fill for users’ data were the top 5 UX issues in
mobile apps (Idealogic, 2021). These problems will try to be rectified in this project’s app.
```
```
All of these design principles will be compiled and used though out the project, especially in the
design phase.
```
#### 2.1.2 UX/UI Summary

```
In summary, it has been found the sheer importance the UX and UI design can have on the success
or failure of an app. The project will therefore use the following points when designing and
implementing the app to maintain a high level a UX and UI design:
```
- Consistency – The design and style must be consistent throughout each app page with the
    same colours and layout used.
- First impression count - The home page/landing screen needs to be friendly, inviting and
    easy to use, as this will set the tone for when the user is using the rest of the app.
- Accessibility / Easy to use – The app needs to be easy to use for all – no one should be
    excluded. The app navigation should flow in a chronological and natural order. The
    navigation should also be in a logical order, labelled correctly and not complicated to
    understand or use.
- User controls – The right buttons for that form/page should be available to the user. Also,
    the option to add user customisation is a possibility too.
- Feedback – The user should not be bombarded with popups, and the right level of user
    feedback, such as after submitting a form, should be found.
- Speed – The app should not be slow to load.
- Simple authentication procedures – The app should have a simple but secure login which is
    fast, reliable and secure, possibly allowing for auto-fill to be used.

```
The following UI Principles should also be considered:
```
- Be Functional.
- Be Reliable.
- Be Enjoyable & Comfortable to use.
- Reduce cognitive load on the user.
- Be Aesthetically pleasing.
- Have a consistent design.


### 2.2 Accessibility and Inclusivity (EDI)

```
It is absolutely critical that the app is accessible for all and does not discriminate against anyone. “An
inclusive app puts people first by prioritising respectful communication and presenting content and
functionality in ways that everyone can access and understand” (Apple, no date). Apple's statement
of inclusivity is a must for any app being developed today. It would be wrong for an app to stop
people from using it if they wish to do so. Also, anything offensive should not be included in the app
as this is simply morally wrong. Goodfirms state that inclusivity in app development is “the need of
the hour” (Watson, no date), and with the aim of the app to help everyone who needs one to get a
GP appointment, it is therefore critical that it is accessible to all of society as much as possible.
```
```
Additionally, a universal design should be implemented to allow the majority of people to use the
app. Vendome et al. (2019) highlighted seven principles to universal design. All seven are critical in
this project and show that a universal design should:
```
- Equitable use
- Be “flexibility in use”
- Be “simple and intuitive [to] use”
- Have “perceptible information”
- Lower “tolerance for error”
- “Low physical effort”
- “Size and space for approach and use”

```
This research has shown how the app needs to be informative and functional but accessible and
usable by as many as possible.
```

### 2. 5 Patient No-Shows

```
Patient no-shows are costing the NHS millions of pounds each year and wasting thousands of much-
needed GP appointments. In 2019, more than 15 million were missed, costing £216 million (NHS,
2019). In 2021/22, around 6.4% of booked appointments were missed – around 650,000 per month.
(NHS England, 2023). This shows how the problem is a continuing issue and not just an issue caused
by the covid-19 pandemic.
```
```
A paper found that “over 40% of individuals who missed an appointment said that they forgot the
appointment” (Neal et al. , 2005). The same paper found that no-shows were caused the most by
practice factors and their [patients] own forgetfulness as the main reasons for doing so” (Neal et al.
2005). Furthermore, in the USA, “3.6 million people miss or delay appointments due to
transportation issues” (Tšernov, 2022).
```
```
Possible solutions to these issues could include reminding appointments by text or phone, charging
for missed appointments, or more flexible ways for patients to book appointments. In late July/Early
August 2022, it was widely reported in the UK that our now prime minister Rishi Sunak would charge
£10 fine for an NHS no-show (Malnick, 2022)
```
```
Therefore, a patient booking system could be helpful as over a quarter of participants in a 2005
paper by Neal et al. said “that they tried very hard to cancel the appointment or that it was at an
inconvenient time”. An on-the-go patient booking app could easily rectify this.
```

## 3. Methodology

### 3.1 Project Methodology

```
This project has chosen to follow the agile methodology and in particular the scrum methodology.
This will allow for the project to be broken down into sprints allowing for specific sections and
features to be completed at a time. Traditionally, the waterfall methodology would have been used
with its iterative approach to the product life cycle, competing each stage before moving onto the
next. However, this approach is unsuitable for this project as it is fast-paced, with requirements
being continuously added throughout most of the project. By implementing agile as the project
methodology, it allows for the project to be more manageable and is known to lead to a higher
quality of end product. With scrum, testing is carried out throughout development, allowing bugs to
be spotted early and therefore fixed early, which can help reduce delays later on in the project.
Using agile also ensures that key functionality of the app is built and tested first before other minor
features are completed ensuring a minimal viable product is completed. Agile is also more equipped
to deal with rapidly changing IT Projects, with “71% of companies use Agile approaches” (Djurovic,
2022). Figure 1 shows how the scrum methodology works.
```

#### 7.2.2 Login and Registration

```
All users must have an account to access the app. Therefore, new users can register through the
login page and existing users can login to their accounts and reset their password if needed. This has
been implemented through the use of various input forms and stored variables.
```
```
On registration, the user is asked to enter account details across two pages with temporary variables
set, which the second page (security questions) can then use and create the necessary records in the
database. The user's name, email and phone number is taken and a new record in the User table is
created with a unique user_ID generated. The username and password entered is taken and a new
account in the login table is created. The created userID is then passed to the login table to link the
accounts together. The database can later use the security answers collected and stored to check
against when users are resetting their account. Validation and verification occurs while the user is
creating their account such as checking for existing accounts with the same username and ensuring
users confirm their typed passwords.
```
```
Users can login to their account as a match is looked for in the database. The password field is case-
sensitive. If a match is found, the userID for the match is stored and then used across the app. On
login, further variables such as user role, email and birthday are pattern-matched in the database
and then stored locally in PowerApps. On success, all temporary variables are reset. If user details do
not match, an error message is shown.
```
```

```

### 7.4 Technical Challenges

```
Throughout the implementation stage, a few issues did arise. One main issue which occurred was
the embedding of the Power BI report into Power Apps. It was found that the report could only be
embedded one page at a time with limited interactive functionality. Because of this, navigation
between the pages and the use of the filter button was not working. After much research through
reading forums, the Microsoft help pages and consulting with peers, it was discovered that only
premium or pro Power BI accounts could embed a full report with full interactivity. It was also
discovered due to university licensing constraints only a free account is available to students. After a
lengthy discussion with the universities IT department, a free trial was requested but denied,
resulting in the loss of this functionality via the app. It is still available via the Power BI file, however.
```
```
Furthermore, Due to security issues when accessing the university’s S L Server, direct query import
of the live app data was not possible. Instead, the data must be manually refreshed. This would have
bi-passed the need to keep refreshing the data and creation of a second data model inside Power BI.
```
```
Additionally, when a new user is created, their user ID links their login with their account in the
database. However, when this was being implemented, the correct user id was not being passed into
the login table. This meant that the account and login details were not linked correctly, allowing for
issues to occur when users tried to login or create an account. This was fixed by consulting forums
and experimenting with code in PowerApps. To fix the issue, the user's user ID and username are
now temporarily stored while creating an account, which is later used to link the login record with
the user account record in the database.
```
```
Also, the developer faced issues when implementing the symptoms drop-down box. At first, the user
was able to select options from this list during the initial booking but went they went to edit; the
changes were not being saved to the database. After consulting with a tutor, it was discovered a
simple spelling mistake of ‘symptons’ instead of ‘symptoms’ had been made in the update function.
```
